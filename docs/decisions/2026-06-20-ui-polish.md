# 2026-06-20 · UI polish(画布 chrome + 设置 hydration + 排版)

> 用户报:"主要是先修下 ui 吧,现在有溢出问题,主要是画布哪里,然后文字排版不舒服,设置里功能调节有的还没搞好(主题色什么的)"。
> 本轮只修前三个真问题;主题色/字体可调/间距密度明确排除,留给下一轮 UI 重设计或 settings-extras spec。
> tag 建议 `v0.22.0-ui-polish`(环境非 git tag 友好,未实际打 tag)。

## 一句话

画布三层 UI 折叠为两层(canvas/page.tsx 删 3 个冗余节点 + tldraw `components` prop 屏蔽自带 chrome + canvas `.page` 用 `--app-menu-height` 避免底部裁剪)+ 设置页 "1 error" 真 bug 修复(theme inline script 引发的 hydration mismatch,`<html>` 加 `data-theme="light"` + `suppressHydrationWarning`)+ inbox/archive 卡片排版 polish(CJK 断字规则 + grid 列宽 280→320 + 字重 500→600 + 行间距 < 列间距)。零数据/接口改动,3 个 commit 闭环。

## C1–C4 · 画布 chrome

**根因**:`canvas/page.tsx` 自己渲染了 `<Toolbar region="canvas">`,含:面包屑(`cy's stift / Canvas / [切换器]`)+ 3 个画布管理按钮 + `double-click to create · drag to place` hint + 卡片计数 `<Tag>` + `← home` link + snap + 缩放。tldraw v3 自带菜单/缩放/share **未被禁用**,两套 UI 在顶部 100px 内叠加,中文按钮 wrap 后被压成竖排。

**fix**(`cc914a5`):
- `canvas/page.tsx`:删 `<span className="hint">` + `<Tag>` + `<Link className="crumb-link">` 3 个节点(冗余,信息已由 AppMenu / tldraw 空状态承担);同步删 inline `<style>` 里的 `.hint` 和 `.crumb-link` 规则
- `canvas-editor.tsx`:`<Tldraw>` 加 `components={{ TopPanel/SharePanel/MenuPanel/NavigationPanel/PageMenu: () => null }}`,保留现有 `hideUi`(双保险,即使 components prop 名未来变了 hideUi 也兜底)
- `globals.css`:新建 `:root { --app-menu-height: 56px; }`(v0.15 原本没有,canvas 直接用 `height: 100vh` → 与 sticky AppMenu 50px 重叠)
- `canvas/page.tsx` inline `<style>`:`.page { height: 100vh }` → `.page { height: calc(100vh - var(--app-menu-height)) }`

**决策**:保留页面 Toolbar(画布管理 = 域逻辑,我们控制;i18n 友好)而非全用 tldraw 自带菜单(需 tldraw-i18n 依赖,YAGNI)。

## C5 · 设置 1 error

**根因**(实测 console stack):
- `<html>` SSR 渲染时无 `data-theme` 属性
- inline `<script>` 在 `<head>` 同步执行,从 localStorage 读 `theme` 后 `setAttribute('data-theme', resolved)`
- React hydration 比较 SSR HTML vs 客户端首次渲染:SSR 无 `data-theme`,但 inline script 已在 React commit 前改了 DOM(有 `data-theme`)
- 报 hydration mismatch → "1 error" 红标

**fix**(`9325cca`):
- `<html>` 加 `data-theme="light"`(SSR 默认 light,与 OS 默认对齐)+ `suppressHydrationWarning`(让 React 跳过该节点属性检查,因为 `data-theme` 由 inline script 决定,SSR 无法预测)
- `suppressHydrationWarning` 只跳过该节点属性,不跳过 children,无副作用

**验证**:puppeteer 抓 console,改前 1 个 hydration error + nextjs-portal overlay 出现;改后 0 个 pageerror + 0 个 hydration warning,只剩 favicon.ico 404(无伤大雅)。

## C6–C7 · 排版 polish

**根因**:
- 全局无 CJK + 西文混排断字规则,英文长串在窄列被强行断字
- grid `minmax(280px, 1fr)` 偏小,空列间距过大
- 卡片标题字重 500,与正文 400 对比弱

**fix**(`a1186fa`):
- `globals.css` 新增 `.tile__title` + `.tile__preview` 规则:
  - `word-break: normal; overflow-wrap: anywhere`(CJK 优先在标点处断,西文词内允许 anywhere)
  - `text-wrap: balance`(标题多行宽度平衡)
  - `line-break: strict`(CJK 严格按标点断)
  - `.tile__title` `font-weight: 600`(从 500 加深,拉大与正文对比)
  - `.tile__preview` `line-height: 1.55`(从 1.5 微调,改善密度感)
- `inbox/page.tsx` 和 `archive/page.tsx` 的 `.grid`:
  - `minmax(280px, 1fr)` → `minmax(320px, 1fr)`(空列概率降低)
  - `gap: var(--space-3)` → `gap: var(--space-3) var(--space-4)`(行间距 < 列间距,卡片纵向紧凑)

**作用域限定**:断字规则只在 `.tile__*` 类内,不污染 markdown 代码块、URL、`<body>` 全局。

## 关键决策

1. **页面 Toolbar 保留而非用 tldraw 自带菜单**:画布管理(域逻辑)vs 画布引擎(渲染),职责不重叠。tldraw 自带菜单是英文无 i18n,且引入 tldraw-i18n 是新依赖。
2. **`<html suppressHydrationWarning>` 而非删 inline script**:inline script 是 dark-mode 不闪的关键(spec §5.6),删了 dark mode 用户会看到 light 闪烁。suppressHydrationWarning 是 Next.js 推荐方案,作用域精确。
3. **globals.css + tile class 而非 page-scoped CSS**:inbox/archive/trash 三页共享 `.tile__*` 类名,改一处生效全部。inline style 优先级问题靠 globals.css 不与 inline 冲突的属性(断字规则 inline 没设)。
4. **v0.18/19/20/21 决策档(中英 i18n / 搜索 / refactor ABC)代码丢弃,决策档保留**:本轮 reset --hard 抹掉了 working tree 里的 v0.18+ 代码改动,代码从 v0.15 基线重启。决策档(supersearch-canvas-preview.md / i18n-bugfixes.md / refactor-abc.md / audit-bugfixes.md)保留作为未来重做依据。本轮 ui-polish 严格不依赖任何 v0.18+ 特性。

# v0.22.0-ux-polish · UX walkthrough 修复(2026-06-20)

> plan 完成后做了一轮 puppeteer-driven UX walkthrough,发现 5 个真 bug,集中在一个 commit `e8a8da4` 修复。

## 一句话

修:CJK 断字仍然把"中文"拆成"中"+"文"(plan 改的 `word-break: normal` 不够,改为 `keep-all`)/ trash 进页弹空 Modal(`!== undefined` 应为 `!= null`)/ canvas 空状态无提示(plan T1 删 hint 后没替代)/ card detail 空 body 显示 "(no body)" 占位文案 / canvas edit card 标签间距过大 / inbox active tab 字重过弱。

## 修复明细

### 1. CJK 卡片标题断字(回归 T3)

**现象**: `word-break: normal` 在 Chrome 上对 CJK 序列不保持整体,仍然在字符间断。

**修法**: 改用 `word-break: keep-all`(CJK 序列作为不可断单元) + `overflow-wrap: anywhere`(允许西文长词在任意位置断) + `text-wrap: balance`(多行宽度平衡)。删 `line-break: strict`(Chrome 不支持且会让 CJK 字符间断)。

### 2. trash 页 Modal 关不掉

**根因**: `Modal open={confirmingCard !== undefined}`,而 `confirmingCard` 初始为 `null`,`null !== undefined` 恒为 true。

**修法**: 改 `!= null`(同时排除 null 和 undefined)。

### 3. canvas 空状态无提示(plan T1 副作用)

**现象**: plan T1 删了原 hint 文本 `double-click to create · drag to place`,canvas 空状态只剩 dot grid 背景,新用户不知道怎么用。

**修法**: 在 `.cv-host` 内加 `.cv-empty` overlay,只在 `onCanvas === 0` 时显示。内容:"EMPTY CANVAS" eyebrow + 提示文字。`pointer-events: none` 保证双击仍穿透到 tldraw 表面创建卡片。

### 4. card detail 显示 "(no body)" 占位文案

**根因**: `MarkdownBody` 在 `source.trim()` 为空时显示 `<p class="md--empty">(no body)</p>`。

**修法**: 改为直接 `return null`(空 body 不渲染任何东西)。如果用户想看"这里什么都没写"——那就是什么都没写,不该用占位文案提示。

### 5. canvas Edit card Modal 标签间距过大

**现象**: Modal 标题("Edit card")与 first child (`.cd__meta` 或 `.cd__field`)之间有 ~24px padding,内部 label 与 input 之间又叠加 `.cd gap=space-3`(12px),合计显得松散。

**修法**:
- `.cd > :first-child { margin-top: calc(-1 * var(--space-2)); }` 让第一个子元素贴紧 Modal title
- `.cd__field { gap: var(--space-1); }` label 与 control 4px 紧贴(从 12px 收紧)

### 6. inbox active tab 字重过弱

**现象**: `ACTIVE` tab 选中时只换了颜色和下划线,字重仍 500(普通字重),与 `ARCHIVED` 对比不够强。

**修法**: `.tab--active` 加 `font-weight: 600`。

## 已知遗留(明确 out of scope)

- **Soft-delete 按钮红色填充混淆**: `danger` variant 是设计系统定义的红填充,但产品语言里红色 = Capture(主行动),同一颜色表示两种语义冲突。这是更深的设计问题,需要重新定义 variant 语义或加 hover tooltip。**留给 UI 重设计阶段。**
- **Archive tab 空状态文案不准确**: "Archived cards show up here. Unarchive to bring them back to the inbox" —— 这文案说"inbox",但用户在 archive 页(不是 inbox archived tab)看到时反而合理。需要分两处文案。**留给文案 polish。**
- **Capture placeholder 红字**: 灵感标题 placeholder 用 `var(--color-red)`,在 dark mode 下对比度差。**留给 design polish。**
- **favicon.ico 404**: 浏览器自动请求,无 favicon 文件。**不在任何 plan 范围内。**
