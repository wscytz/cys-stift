## 3. 画布组件技术路线梳理:tldraw 依赖评估与自研替代方案

> 我已将你零散的想法整理成结构化的技术决策框架,涵盖核心问题拆解、可行性分析和行动路径,
> 方便你后续与团队讨论:

**处置**:整理为 `docs/plans/2026-06-22-canvas-strategy-tldraw-vs-self-build.md`(proposal);
三开源项目调研见 `2026-06-22-canvas-research-drawio-archdiag-affine.md`。

### 一、核心问题总览

四个相互关联的技术决策:

1. **自研画布组件替代 tldraw 的可行性** — 是否要做、何时做、做到什么程度
2. **当前 tldraw 依赖深度评估** — 明确绑定程度,判断迁移成本
3. **基于文本描述语言的画布设计** — 从 AI 能力出发,重新定义画布底层数据模型
4. **底层语言基座更换可行性** — 评估从现有技术栈切换到更开源方案的收益与风险

### 二、tldraw 依赖程度与远期规划评估

所有决策的基础。先做现状盘点。

**2.1 功能依赖清单(按依赖强度排序)**
- 强依赖:核心绘图(形状/连线/文本/自由画笔)、无限画布、缩放平移、图层、撤销重做、导出
- 中等依赖:多人协作、快捷键系统、自定义形状、插件生态
- 弱依赖:UI 组件库、主题系统、默认工具栏布局

**2.2 数据与集成依赖**
- 是否直接用 tldraw 原生 JSON 存?
- 业务逻辑是否与事件系统(onChange / onShapeCreate)深度耦合?
- 基于 tldraw API 的自定义功能模块数量 + 复杂度?

**2.3 商业与法律依赖**
- 当前 tldraw 版本开源免费还是商业收费?(项目用 `3.15.6`,Apache-2.0 + 商标条款)
- 收费版成本 + 涨价风险
- 许可证对分发/部署的限制

**2.4 远期规划建议**

| 阶段 | 时间 | 策略 | 核心动作 |
|---|---|---|---|
| 短期(0-6 月) | 继续用 tldraw | 深挖现有功能 | 封装抽象接口层,隔离业务代码与 tldraw API |
| 中期(6-18 月) | 部分自研 | 针对 AI 核心需求开发自定义模块 | 基于文本描述语言实现 AI 生成画布核心,作为 tldraw 插件 |
| 长期(18 月+) | 全面自研或混合 | 按业务决定是否完全替代 | 若 AI 画布成核心竞争力,逐步替换;否则保留作基础层 |

### 三、自研可行性

**优势**:AI 深度融合 / 功能定制化 / 成本可控 / 数据主权。
**挑战**:开发成本高 / 功能差距大 / 维护负担重 / 团队能力要求(Canvas/SVG、图形学、性能)。

**可行路径(渐进式)**
1. 封装抽象层(Canvas Abstraction Layer)—— 与 tldraw 无关的通用接口;成本最低。
2. 开发 AI 专属画布模块 —— 基于文本描述语言,作为 tldraw 插件;验证描述语言可行性。
3. 最小可行画布(MVP)—— 业务必需核心绘图;支持导入导出 tldraw 格式;特定场景试点。
4. 全面替代或混合 —— 据 MVP 反馈决定。

### 四、基于文本描述语言的画布设计

自研核心差异化 + 与 AI 结合的关键切入点。

**4.1 XML vs JS/JSON**:XML 结构化强、AI 友好,但前端需解析;JS/JSON 前端原生、工具链丰富。

**4.2 推荐**:JS/TypeScript 严格类型 + JSON 存储为主描述语言;XML 为辅助交换格式(大模型生成复杂嵌套结构时 XML 准确率更高)。

**4.3 与 AI 结合设计要点**:语义化设计(`type:'process'` 而非 `'rectangle'`)/ 增量更新 / 版本控制 / 提示词工程。

### 五、更换底层语言基座

**动机**:避免商业许可风险 / 更活跃社区 / 更好性能或功能 / 与现有栈更好融合。

**主要开源替代**

| 方案 | 技术栈 | 核心优势 | 主要劣势 |
|---|---|---|---|
| Konva.js | Canvas | 功能完善,性能好 | 无限画布要自实现 |
| Fabric.js | Canvas | 老牌,功能强大 | 架构老,性能一般 |
| PixiJS | WebGL | 性能极佳 | 基础绘图要自封装 |
| Excalidraw | React+Canvas | 手绘风,开源免费 | 自定义形状弱 |
| Draw.io | 多种 | 功能极完善 | 代码复杂,二次开发难 |

**决策建议**:① 未完成抽象层前不直接换基座;② 可先在 AI 专属模块试用新基座;③ 优先 React 兼容方案;④ 若主因是避免商业收费,Excalidraw 值得优先考虑。

> **项目约束**:根 `CLAUDE.md` 硬禁「重新选型—— 要改先写 ADR」。任何更换必须先走 ADR + spec 审查。
