## 背景

review §🟠 UX 洞 #2(原文):**"'Send to canvas' 后卡从 inbox 消失(spec §6.11 `listInbox` 过滤 canvasPosition),且无'从画布拿回 inbox'反向动作"**。

inbox CardDetailModal 已有"Send to canvas"按钮(Phase 6.5c)→ 卡上画布后无法拿回 → 需 unarchive + unarchive-to-canvas-reverse 的迂回路径,UX 摩擦大。

## 决策

| 决策 | 选项 | 选择 | 理由 |
|---|---|---|---|
| 入口 | inbox 页回(逻辑反) / **canvas CardDetailModal 加按钮** | **canvas 加** | canvas 是卡当前位置,放"返回"按钮自然;inbox 显示逻辑不需改(`listInbox` 已排除 canvasPosition 卡,`canvasPosition` 一清自动回来)。 |
| domain 方法 | 复用 `update` / **新 `removeFromCanvas`** | **新方法** | 卡片 lifecycle 字段(archived / deletedAt / canvasPosition)不通过通用 `update` 改,与 `moveToCanvas` / `softDelete` / `restore` 对称;boolean 返回 + idempotent 与 `hardDelete` / `restore` 一致。 |
| Button 位置 | 总是显示 / **仅当 `card.canvasPosition`** | **conditional** | 兜底:已是 inbox 的卡不会到画布,但若某种 edge case 出现(canvasPosition 为 undefined),按钮不显示避免无效点击。 |
| inbox 显示 | 改 inbox filter / **不动,依赖 `listInbox` 已排除 canvasPosition** | **不动** | spec §6.11 已定义 `listInbox` 排除 canvasPosition,`removeFromCanvas` 清字段 → 卡自动回 inbox,零 inbox 端代码。 |

## 关键工程决策(总结)

- **canvas 加按钮(非 inbox)**:画布是当前位置,"返回"按钮自然;inbox 零改动。
- **新 `removeFromCanvas` 而非 `update`**:lifecycle 字段对称,boolean return + idempotent 与 `hardDelete` / `restore` 一致。
- **不动 inbox 端**:`listInbox` 已排除 `canvasPosition` 卡(spec §6.11 已定义),`removeFromCanvas` 一调即生效。
- **conditional 按钮**:`card.canvasPosition && onSendToInbox` 兜底,避免无效按钮。

## 显式留后

- send-to-canvas 后,**archive 内的卡也支持 send-back**(目前 listInbox + canvas 二选一,archive 内已 archived 的卡上画布逻辑见 review)
- archive tile 接 detail 时未接 "Send to canvas" / "Send back to inbox"(archive 上下文卡不需上画布,符合当前设计)
- "Send to canvas" 后卡从 inbox 隐藏的反向 UX 已通过 send-back 解决,无需新 prompt
