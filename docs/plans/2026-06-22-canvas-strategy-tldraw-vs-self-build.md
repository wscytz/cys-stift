# 画布技术路线梳理:tldraw 依赖评估与自研替代方案

> 来源:开发者反馈 2026-06-22。这是**讨论框架(proposal)**,不是已定决策。
> 供后续与团队讨论。任何行动需先走 ADR + spec 更新(spec 冻结,改要五轮审查)。

## 一、核心问题总览

四个相互关联的技术决策:

1. **自研画布组件替代 tldraw 的可行性** — 是否要做、何时做、做到什么程度
2. **当前 tldraw 依赖深度评估** — 明确绑定程度,判断迁移成本
3. **基于文本描述语言的画布设计** — 从 AI 能力出发,重新定义画布底层的数据模型
4. **底层语言基座更换可行性** — 评估从现有技术栈切换到更开源方案的收益与风险

## 二、tldraw 依赖程度与远期规划评估

所有决策的基础。先做现状盘点。

### 2.1 功能依赖清单(按依赖强度排序)
- **强依赖**:核心绘图(形状/连线/文本/自由画笔)、无限画布、缩放平移、图层、撤销重做、导出
- **中等依赖**:多人协作、快捷键系统、自定义形状、插件生态
- **弱依赖**:UI 组件库、主题系统、默认工具栏布局

### 2.2 数据与集成依赖
- 是否直接用 tldraw 原生 JSON 格式存储?
- 业务逻辑是否与其事件系统(onChange / onShapeCreate)深度耦合?
- 基于 tldraw API 的自定义功能模块数量 + 复杂度?

### 2.3 商业与法律依赖
- 当前 tldraw 版本是开源免费还是商业收费?(项目用 tldraw `3.15.6`,Zenjoy/Apache-2.0 + 商标条款 — 需确认分发场景)
- 收费版的成本区间 + 涨价风险
- 许可证对分发/部署的限制

### 2.4 远期规划建议

| 阶段 | 时间 | 策略 | 核心动作 |
|---|---|---|---|
| 短期(0-6 月) | 继续用 tldraw | 深挖现有功能,不重复造轮子 | 封装统一抽象接口层,隔离业务代码与 tldraw API |
| 中期(6-18 月) | 部分自研 | 针对 AI 核心需求开发自定义模块 | 基于文本描述语言实现 AI 生成画布核心,作为 tldraw 插件运行 |
| 长期(18 月+) | 全面自研或混合 | 按业务决定是否完全替代 | 若 AI 画布成核心竞争力,逐步替换 tldraw 核心;否则保留作基础层 |

## 三、自研画布组件替代 tldraw 的可行性

### 3.1 自研核心优势
- **AI 深度融合**:从底层设计支持 AI 生成/理解/编辑的画布数据模型,而非在 tldraw 之上打补丁
- **功能定制化**:完全掌控演进路线
- **成本可控**:避免长期商业收费(尤其用户量增长后)
- **数据主权**:完全自主的数据格式

### 3.2 自研主要挑战
- **开发成本高**:基础绘图引擎、性能优化、跨平台兼容需要大量投入
- **功能差距大**:tldraw 多年迭代,自研短期难达同等水平
- **维护负担重**:持续修 bug / 优化 / 跟进浏览器更新
- **团队能力要求**:Canvas/SVG、图形学、前端性能优化专精工程师

### 3.3 可行的自研路径(推荐渐进式)
1. **第一步:封装抽象层(Canvas Abstraction Layer)** — 定义一套与 tldraw 无关的通用画布接口;所有业务代码通过抽象层调用。成本最低,即使不自研也提升可维护性。
2. **第二步:开发 AI 专属画布模块** — 基于文本描述语言实现 AI 生成/解析核心,作为 tldraw 插件运行,与现有功能无缝集成。验证文本描述语言的可行性。
3. **第三步:最小可行画布(MVP)** — 实现业务必需的核心绘图;支持导入导出 tldraw 格式保证数据迁移;在特定场景(如 AI 生成的简单流程图)试点。
4. **第四步:全面替代或混合** — 据 MVP 反馈决定是否继续完善;也可混合(基础绘图用 tldraw,AI 相关用自研)。

## 四、基于文本描述语言的画布设计

自研的核心差异化 + 与 AI 结合的关键切入点。

### 4.1 XML vs JS 格式对比

| 维度 | XML | JS/JSON |
|---|---|---|
| 结构化程度 | 强,标签语义明确 | 中等,键值对 |
| AI 友好性 | 较高,大模型对 XML 理解/生成较强 | 也不错,但易语法错 |
| 前端易用性 | 较差,需解析成 JS 对象 | 极好,原生支持 |
| 扩展性 | 好,属性 + 子标签 | 好,键值对 |
| 可读性 | 较好,标签结构清晰 | 一般,嵌套深时差 |
| 工具链支持 | 一般 | 极其丰富 |

> 项目现状参考:`docs/decisions/2026-06-21-ai-context.md` 的 DSL parser 已走类 JS/JSON 方向(`@pos(x,y)` / `at (x,y)` / `size WxH`)。本节是更底层的画布数据模型讨论,不是 DSL 本身。

### 4.2 推荐方案:JS 格式主描述语言,XML 辅助交换
- **主描述语言**:TypeScript 严格类型,JSON 存储传输
  ```typescript
  interface CanvasElement {
    id: string
    type: 'rectangle' | 'ellipse' | 'text' | 'line' | 'group'
    position: { x: number; y: number }
    size: { width: number; height: number }
    style: { fill: string; stroke: string; strokeWidth: number }
    children?: CanvasElement[]
    metadata?: Record<string, unknown> // AI 生成相关元数据
  }
  ```
- **辅助交换格式**:XML ↔ JSON 双向转换;用于系统集成;大模型生成复杂结构时 XML 嵌套准确率更高。

### 4.3 与 AI 结合的设计要点
1. **语义化设计**:加入语义信息而非纯视觉。`{ type: 'process', label: '用户登录', connections: [...] }`(语义类型,不只是 `rectangle`)。
2. **增量更新**:支持局部修改的数据模型,方便 AI 改画布局部。
3. **版本控制**:内置版本机制,支持 AI 多版本对比 + 回滚。
4. **提示词工程**:针对我们的描述语言优化提示词,提高生成准确率。

## 五、更换底层语言基座的可行性

### 5.1 动机
避免商业许可证风险 / 更活跃社区 / 更好性能或功能 / 与现有栈更好融合。

### 5.2 主要开源替代方案

| 方案 | 技术栈 | 成熟度 | 核心优势 | 主要劣势 |
|---|---|---|---|---|
| Konva.js | Canvas | 高 | 功能完善,性能好,社区活跃 | 无限画布要自己实现 |
| Fabric.js | Canvas | 高 | 老牌,功能强大 | 架构老,性能一般 |
| PixiJS | WebGL | 高 | 性能极佳,适合复杂图形动画 | 基础绘图要自己封装 |
| Excalidraw | React+Canvas | 高 | 手绘风格,多人协作,完全开源 | 自定义形状能力弱 |
| Draw.io | 多种 | 极高 | 功能极完善,多图表 | 代码复杂,二次开发难 |

### 5.3 匹配度评估框架
核心功能匹配 / AI 集成难度 / 性能 / 社区活跃度 / 文档质量 / 迁移成本 / 长期维护成本。

### 5.4 决策建议
1. **不建议**在未完成 tldraw 抽象层封装前直接换基座。
2. 可先在 AI 专属画布模块试用新基座,验证可行性。
3. 优先考虑与现有栈(React)兼容好的方案。
4. 若主要动机是避免商业收费,**Excalidraw** 值得优先考虑(完全开源免费,功能与 tldraw 接近)。

> **项目约束提醒**:根 `CLAUDE.md` 硬性禁止"重新选型(换框架/ORM/数据库)—— 要改先写 ADR"。本节讨论的任何更换**必须先走 ADR + spec 审查**,不能直接动手。

## 六、下一步行动建议

1. **一周内**:完成 tldraw 依赖程度盘点,输出详细依赖清单。
2. **两周内**:设计并实现画布抽象层,隔离业务代码与 tldraw API。
3. **一个月内**:基于 JS/TypeScript 定义画布文本描述语言的初步规范,实现简单的 AI 生成 + 解析。
4. **两个月内**:完成 2-3 个开源替代方案的技术调研 + 原型验证,输出对比报告。
5. **季度末**:据以上结果,制定明确的中长期技术路线图。

## 与项目现状的对照(2026-06-22)

- **AI 画布已落地**:`docs/decisions/2026-06-21-ai-context.md`(allowlist + canvas snapshot)+ `2026-06-21-ai-layout.md`(DSL parser + applyLayout)。当前的 AI DSL 就是"文本描述语言"的雏形 —— 走 JS/JSON 风格。
- **抽象层未做**:业务代码(canvas-binding / apply-layout / relation-* / auto-relate)直接调 tldraw API,无中间抽象层。第二步(抽象层)是最低成本 + 最高杠杆的起点。
- **冻结约束**:spec §3.4/§6.x 定了 tldraw;换基座属"重新选型",需 ADR + spec 五轮审查。
