## 二、打磨 vs 修缝(核心判据)

判断标准**不是**"还有没有缝可修",而是**"这一步是否推进了产品的核心承诺 / 真实用户是否会碰到"**。

### 🟢 是打磨(值得做)— 满足任一:
- **真实反馈驱动**:你(或任意真实用户)在正常使用中碰到 / 会碰到它
- **阻断或明显劣化核心动线**:捕获 → 养 → 沉淀 的某一步
- **真 bug**:数据丢失 / 崩溃 / 卡死状态 / 错误行为——**永远修**,不论来源
- **可衡量地推进差异化卖点**:转义(画布↔文字双向)/ 语义关系签名
- **发布门**:某个发布版本显式 gate 在它上面(如签名、关键路径手测通过)

### 🟡 是缝(默认跳过,除非极便宜)— 满足任一:
- 只有盲审才发现,**没用户会注意到**
- 理论的 / 难以复现的("理论上可能…")
- 边际的 / 纯防御性的("可能对某人某处有帮助")
- **为一致而一致**,无用户可感知影响
- 需要重新选型 / 大重构 → 走 ADR,不在打磨期顺手做

**一句话**:真实用户碰得到的真问题 = 打磨;只在审计里存在的理论问题 = 缝。

## 三、流程(反馈驱动,非盲扫)

### 主回路(默认)
```
你手测打包版 → 报反馈(哪别扭 / 哪崩 / 哪不顺)→ triage(套§二判据)→ 修
```
- 反馈可以是一句话("切画布后工具栏点不动""拖文件没反应")。
- triage 时套判据:真 bug / 阻断动线 → 立修;纯偏好 → 进 backlog 看优先级。
- 执行沿用 Ralph 式 subagent 编排(主模型拆+审+commit,subagent TDD),但**由反馈触发**,不主动开审计轮。

### 何时才主动扫审计(非常规)
仅当以下之一,才开 Explore/审计轮:
1. **类目可疑**:某个子系统刚大改,或疑似有数据丢失类风险 → 定向扫该类目
2. **发布前加固**:准备发稳定版 / 打 tag 前,做一轮真 bug 猎 + a11y 复核
3. **差异化卖点相关**:转义 / 语义关系的回归或完整性(核心承诺,值得主动保)

**反模式(不做)**:无具体触发地"再扫一轮找找问题"。这是 §二说的修缝循环。

## 五、退出标准(防无限打磨)

打磨期**不是**无止境的。判定"打磨够了 / 可以发"的条件(满足任一即可停):

1. **发布门达成**:你手测核心动线(捕获 / inbox 编辑 / 画布养 / 归档沉淀 / 导入导出往返)全过,无 HIGH/MED 真 bug → 可发稳定版。
2. **反馈枯竭**:连续一轮手测无新真问题,backlog A 空 → 打磨期收口,等下一轮真实使用再开。
3. **触发重构/选型**:某反馈指向需要重新选型 / 大重构 → 打磨期让位,走 ADR + 新 phase。

**不要**用"backlog B/C/D 全清完"作为退出条件——那些多数是缝,清完它们不产生用户价值。

## 七、怎么用这份文档

- **新会话 / compact 后**:先读这份(判据)+ STATE(状态),再决定干什么。
- **收到反馈**:进 backlog A,套§二判据 triage。
