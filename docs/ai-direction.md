# cy's Stift — AI 方向

> 本文档供**协作者 / 顾问 / 潜在伙伴**了解 cy's Stift 在 AI 上的方向与取舍。
> 专注 AI 这块;整体产品定位见 [`product-brief.md`](product-brief.md)。
> 带**轻度技术**细节(讲为什么这么选、用户能感知什么),不涉及代码实现。

---

## 一句话

cy's Stift 的 AI 是一个**本地优先、用可读文本 DSL 与画布对话、永远在你确认下才动手**的画布副驾。

不是聊天框,不是自动驾驶,不是内容生成器。是"看懂你的画布、用你能读懂的语言提议改动、你说行才落地"的协作伙伴。

---

## 它是什么 / 不是什么

**是**:一个读懂画布结构的副驾 —— 你用自然语言提需求,它输出一段**你能读懂、能直接改、能拒绝**的文本指令(DSL),你在确认门看清楚 before/after 才让它落地,且随时可撤销。

**不是**:
- **不是聊天框**:它的每次回答都带着你当前画布的真实结构(不是无上下文的泛泛而谈)。
- **不是自动驾驶**:从不自动改你的东西,必须过你的确认门。
- **不是内容生成器**:它帮你整理 / 重组 / 连接 / 结构化,但笔记内容还是你自己写。
- **不是云依赖**:核心全本地,离线也能用;AI 是可选增强,不是运行前提。

---

## 三条腿:两个里子,一个壳

cy's Stift 的 AI 差异化不是单点,是一个组合。**两个里子是价值,一个壳把它们递到你手里。** 我们判断:市面工具(云端协作白板 / 视觉知识管理 / 开发者白板 / 本地优先笔记)各有侧重,但**没一家同时做到「本地优先 + AI 能完整操作画布 + 开放可读接口 + 提议确认式交互」**——这个交叉点是我们看到的空位。

### 里子 A —— 转义 / DSL 是 AI 的可读接口

市面多数 AI 工具改你的东西,靠的是**黑箱 API 调用**(AI 决定删哪、改哪,你看不见过程)。cy's Stift 不一样:AI 改画布,输出的是一段**人能读懂的文本指令**(`cys-dsl`)—— "把这张卡移到这儿、用这种箭头连那两张、加个矩形框住这组"。

这意味着:**AI ↔ 人 ↔ 画布三方说同一种语言**。你能看见 AI 想干嘛、能直接改它的提议、能整段拒绝。甚至能把这段 DSL 复制粘贴到 ChatGPT 网页版,让别的 AI 驱动同一张画布——接口是开放的,不锁定我们。

这一条承接产品的核心卖点"转义"(画布能完全文字化),是它在 AI 侧的自然延伸,也是**最独特的一条**。

为什么选 DSL 当接口(而不是让 AI 直接调函数、或直接执行自然语言):

- **人可读、可改、可调试** —— AI 输出不完美时,人能定位问题、能直接手改。
- **容错强** —— 解析器能兜住不完美输出(我们有一整套容错测试);不像函数调用,参数错一个就整体失败。
- **开放、可移植** —— 任何外部 AI 不用学我们的 API,能产出这段文本就能驱动画布。
- **可审计、可追溯** —— 每次 AI 操作都有文本记录,可回放、可撤销。

这是我们判断里**最深的一条护城河**——短期用户未必感知,长期是生态与开放的根基。

### 里子 B —— 本地优先 + 隐私

你的笔记、笔迹、想法**默认不外发**。AI 看到的是**最小化、脱敏的画布快照**——只够它理解结构(哪些卡、怎么连、什么布局),不够复原你的私密内容。媒体、设备信息、被你删掉的卡,从不进 AI 的眼。

更进一步:支持**本地模型(Ollama)**。开启后连那段脱敏快照都不离开你的机器——零成本、零外发、断网也能跑。隐私不是开关,是默认态。

### 壳 C —— Claude Code 式渐进 agent(为什么选这个形式)

交互形态我们刻意选了 **"对话 → AI 用 DSL 提议 → 确认门 → 应用可撤销"** 这个范式。

为什么?因为它是**目前"人与 AI 协作改东西"最好的形式**——代码界(Claude Code、Cursor)已经验证过:AI 提 diff,人 review,确认才合并。我们把它搬到**创意画布**:AI 提"画布 diff",你 review,确认才落地。

A 和 B 是里子(说什么、不说什么),C 是把它们交到你手里的那个壳。没有 C,A 和 B 是一堆价值观;有了 C,它们变成一个你能天天用、用得放心的工具。

**确认门不是一刀切** —— 我们倾向**分级**:整理 / 排版这类无害操作走轻确认(一键应用、不必开确认门深 review),删卡 / 批量改动这类破坏性操作必过强确认门;DSL 文本默认折叠(以可视化 before/after 为主),高级用户可展开直改。未来也可能并存两种节奏:**提议模式**(整理优化,AI 提一个改动你确认)和**生成模式**(从零创作,AI 给一堆方案你挑)—— 创意工作两种都要。

## 今天能做什么

- **统一 AI 对话**:/ask 全屏(偏创造,看所有画布)+ 画布伴侣悬浮(偏整理,当前画布)—— 共享同一段对话记忆,跨 session 持久。
- **AI 排版 / 整理**:聚类(按语义分组)、自动布局(思维导图 / 流程图 / 网格 / 紧凑,四方向可选)、智能关系推荐(本地几何信号 + AI 语义粗筛)。
- **卡片 AI 动作**:总结、改写、翻译——入口永远可见,配好 key 才显能力。
- **RAG 引用**:AI 回答你的提问时,引用具体卡片(`[card #id]`,可点开溯源),不凭空说。
- **手绘识别**:画个箭头 / 方框 → 一键转成规则图形。

## 技术取舍(轻)

每条都对应一个用户能感知的好处:

| 选择 | 不选什么 | 为什么 |
|---|---|---|
| **自研 Canvas 2D 引擎** | tldraw 等现成白板库 | 引擎能独立剥离、画布能完全文字化(转义的前提);首屏体积降 ~470kB |
| **文本 DSL 当 AI 接口** | JSON 操作日志 | 人能读懂、能改、能粘贴到外部 AI 驱动同一画布(开放,不锁定) |
| **provider 随你带** | 锁死某家 AI 服务 | 自带 key(任意 OpenAI 兼容端点:OpenAI / DeepSeek / 火山 / SiliconFlow),或本地 Ollama 零成本零外发 |
| **出错自动重试** | 换更大模型硬砸 | 把解析错误喂回 AI 让它自我修正;用工程兜底,不烧钱烧参数 |
| **静态导出 + 本地存储** | 云端服务依赖 | 桌面应用(Next.js 静态导出 + Tauri),数据本地 SQLite / OPFS,离线全可用 |

一句话:**能用工程和设计解决的,不靠堆模型、不靠云。**

## 方向(时间分层)

### 短期
- 把 **"prompt + 自动重试"跑到天花板**:已实装(parse 错误自动喂回重生成,默认 3 次)。**失败样本采集已落地**——retry 耗尽时记 `parse_failed` 样本(坏输出 + 错误 + 尝试数),进本地 sample-store、设置页可导出。用它驱动后续 prompt 加固,不瞎猜。
- AI 对话入口已统一(/ask + 伴侣共享记忆)。

### 中期
- **指令遵循**更稳(复杂多步意图不丢、不跑偏)。
- 接通 **vision 能力**(已搭骨架、未接线的实验室特性,用户显式开启):AI 看图理解画布布局、图片转 DSL、手写 OCR。默认关,有不可逆确认门。

### 远期(愿景)
- **本地 / 任务专用模型**:为 cy's Stift 的 DSL 生成 + 指令遵循 fine-tune 一个小模型,本地跑。卖点 = 隐私 / 延迟 / 成本 vs 云大模型。**触发条件**(到一条再走):prompt+重试优化到顶仍不够准 / 推理成本或延迟成瓶颈 / 离线本地运行成硬需求。**路线**:用大模型当 teacher、蒸馏出小模型 student,目标是在这个垂直任务上接近大模型效果、本地可跑、快且省。**前置**:先攒足够真实的「意图 → DSL」语料(样本导出已落地),失败案例尤其值钱。

### 诚实的待验证
- **专用模型 vs prompt+重试**,在遵循度 / 延迟 / 体验上谁赢?需要找人做对照用户研究,不拍脑袋。
- OCR / 手写文字识别:已实测现状(浏览器可跑的轻量模型 + 手写准确率)不达标,defer 到模型成熟。

## 不做的边界

- **不自动驾驶**:AI 永不 silent auto-apply —— 用户始终是触发方(确认门有轻重分级,见 §3C,但绝不在你不知情时改东西)。
- **不外发训练**:默认数据不进任何训练;样本导出是用户主动行为,本地优先。
- **不云依赖**:核心全本地,AI 是增强不是前提。
- **不替你创作**:AI 改结构、给建议;笔记内容始终是你写的。
- **OCR / 手写文字**:等模型成熟(已 spike,现状不行)。

## 给读者的几个问题(求反馈)

写出来是想被挑战的。如果你在做 AI / 产品 / 研究,这几个问题我们特别想听你的判断:

1. **CC 式"提议 → 确认"在创意画布(非代码)场景,是否同样成立?** 还是创意工作更适合"AI 直接生成一堆,事后挑 / 调"?代码能 review diff,画布的 diff 人眼读得过来吗?
   *我们倾向*:成立 —— 创意同样要可控感,可视化 diff 比代码 diff 还直观;但需**分级确认**(见 §3C),且探索"生成模式"与"提议模式"共存。想听你对"分级会不会破坏流畅度"的看法。
2. **本地优先 + 隐私 vs 云端更强能力,用户实际买哪个账?** 隐私是卖点还是噱头,取决于谁?
   *我们倾向*:分人群 —— 合规 / 专业用户是硬需求,技术爱好者是价值观认同(早期种子),普通用户是加分项。讲法不同:对普通用户讲"快、流畅、不卡",对专业用户讲"合规、安全、可控";自带 key + 本地模型给选择权是关键。想听你的目标人群判断。
3. **任务专用模型(本地 fine-tune)在 cy's Stift 这种垂直 DSL 场景,值不值得投入?** 有没有更聪明的路径(比如更好的 prompt 工程 + 工具调用)?
   *我们倾向*:现在不值 —— 先 prompt+重试+RAG 跑到顶、攒语料;触发条件到了(见 §6 远期)再走蒸馏。想听你对"有没有更聪明的路径"的判断。
4. **转义 / DSL 作为 AI 接口,在外部协作者眼里(把画布粘到 ChatGPT 驱动)是不是真有价值?** 还是我们自嗨?
   *我们倾向*:真有价值,但分阶段 —— 当前主服务内部协作,成长期开放规范,成熟期建生态;对技术用户价值大,对普通用户感知弱。想听你觉得"现在对外讲 DSL"是不是太早。
