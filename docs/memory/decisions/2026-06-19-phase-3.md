## 关键工程决策

### 1. `CardService.update(id, patch)` 设计

只 patch 业务字段（title / body / type / color / pinned / links / codeSnippets / quotes），不 patch 生命周期字段（archived / deletedAt / canvasPosition）。原因：
- 生命周期字段有专门方法（archive / unarchive / softDelete / moveToCanvas）—— 收敛边界，避免一个 update 方法变成大杂烩
- `updatedAt` 自动 bump，无需 caller 关心

### 2. Markdown 安全双保险

`rehype-sanitize` 默认 schema 拦 `<script>` / `javascript:` / 事件处理器。但 sanitize 不改 `<a>` 的 target/rel，也不限制协议（`data:` / `vbscript:` 等可能绕过）。

加固方案：
```tsx
components={{
  a: ({ href, children, ...rest }) => {
    const safeHref = (http | https | 相对) ? href : '#'
    return <a href={safeHref} target="_blank" rel="noopener noreferrer" {...rest}>{children}</a>
  }
}}
```

这样即使 sanitize 配置漂移，UI 层也兜底协议白名单 + 链接安全属性。

### 3. `react-markdown` v9 peer dep 警告

`next@15.0.3` 官方声明 peer `react@^18.2.0 || 19.0.0-rc-66855b96-20241106`，我们用 `react@19.0.0`。pnpm install 报 unmet peer warning，但运行时实测无碍（puppeteer 自动化所有断言通过）。考虑后续升级 next 到 15.1+，但不在 Phase 3 范围。

### 4. Detail Modal 不暴露多媒介编辑

Phase 3 plan §3 P3-T3 明确："编辑模式：标题/正文可改，'保存'调 `service.update`"——只覆盖这两个字段。Detail Modal 内显示链接 / 代码 / 引用（只读），底部显式提示"Editing links/code/quotes is intentionally not exposed here (Phase 3 MVP)"。

原因：
- 多媒介编辑需要链接列表 / 代码 / 引用三套 draft UI → 在 Modal 里塞下太挤
- Phase 3 MVP 简化：用户想改多媒介就 archive + 重新创建
- Phase 3.5 增量：把 Create form 的 ListEditor 拆出来共享

### 5. toolbar Tag 颜色随 view 切换

`<Tag color={view === 'inbox' ? 'red' : 'blue'}>{对应 view 的 count}</Tag>` —— 视觉上直接对应所选区域，避免 "ARCHIVED | 3" 但视觉是红的迷惑（早期版本的 bug，已修）。

### 6. 持久化通过 Phase 2 的 db-client 验证

`useDb()` 返回 reactive snapshot → `service.create/update/archive/softDelete` 触发 `_cards` 替换 → snapshot 引用变化 → `useSyncExternalStore` rerender → localStorage 同步落盘。无需新基础设施。
